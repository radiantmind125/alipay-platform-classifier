# 真值锚点采集方案(不做人工复核，也能知道识别率到底多准)

> 目的：现在的识别率是"拿自己生成的弱标签考自己"，是循环的、不可信的(只有 65 张人工核对)。要一个**非循环的真值参照**，才能真正知道多准、才能定"自动判 vs 转复核"的线、才能防每次改模型悄悄掉点。
>
> **关键：这完全在"不做人工复核"的约束内。** 约束禁的是"让人去逐张判存量图是真是假/是什么系统"(运营做不了、对好假图也会判错)。本方案的标签**不是人判图**，而是**"这台机器本身是什么系统"**——采集，不是复核。

---

## 一、采什么

用**已知系统的真机**，截支付宝"转账成功"页。覆盖这几个维度(安卓状态栏各厂商差异大，务必铺开)：

- **系统**：iOS + 安卓；安卓要覆盖 华为 / 小米 / OPPO / vivo / 三星 等主流厂商(状态栏样式各不同)。
- **形态/分辨率**：刘海 / 灵动岛 / 直屏都要；短边覆盖 720 / 1080 / 1440 以及各 iPhone 分辨率；**特意包含一些被裁过/缩放过的**(残差区正是模型的弱项)。
- **深色 / 浅色**模式各来一些。
- **翻拍正样本**：部分图再对着屏幕拍一遍，得到"出处已知的翻拍图"，用来喂翻拍检测。

## 二、采多少(分层，约 1600–2000 张)

统计依据：要声称某层 ≥99%，观察到 0 错时 95% 上限≈ 3/n，故 ~300 张；要给残差区一个 ±3%/95% 的精度区间，~400 张。

| 层 | 目的 | 建议张数 |
|---|---|---|
| iOS·分辨率命中(刘海/灵动岛) | 验证 Tier-0 对 iOS 真有 ~99% | 300 |
| iOS·残差(缩放/裁过/老机型) | 测 CNN 在 iOS 残差区**真实精度** | 400 |
| 安卓·分辨率命中(720/1080/1440) | 验证 Tier-0 对安卓 | 300 |
| 安卓·残差(非标准分辨率/异形比例) | 测 CNN 在安卓残差区 | 400 |
| 翻拍(iOS+安卓各半) | 出处已知的翻拍正样本 | 200 |
| **合计** | | **~1600** |

一个人几天点机可得，和"标 12 万"不是一个量级。也可以让**几位同事各用自己手机截几张**(系统已知)，很快能凑齐。

## 三、怎么标(零人工判断)

标签 = **采集时这台机器的系统**，不看图内容。用一份**清单(manifest)**记录每张图的出处即可。建议一个 CSV：

```csv
文件名,系统,厂商机型,分辨率,深浅色,是否翻拍
prov_0001.png,ios,iPhone13,1170x2532,dark,orig
prov_0002.png,ios,iPhone13,1170x2532,dark,photo
prov_0101.png,android,MI11,1080x2400,light,orig
prov_0102.png,android,Huawei_P40,1080x2340,light,orig
```

- `系统` 只有 ios / android 两个值，来自设备本身，**不是人判的**。
- `是否翻拍`：orig=原始截图，photo=对屏幕拍的。
- 文件名随意，但要和 CSV 对得上。图片和这份 CSV 一起给我就行。

## 四、怎么用(拿到后我来做)

这批一次能产出四样东西——都是现在拿不到的：

1. **CNN 在残差区的真实精度 + 置信区间**(现在完全是黑的)。
2. **弱标签噪声率**：把分辨率规则跑在这批上，`不一致/总数` 就是弱标签的噪声率 = CNN 精度的天花板。
3. **标定不确定阈值**：用可靠性曲线(reliability diagram)把"自动判 vs 转复核"的线定在你要的精度上(比如"精度≥99.5% 才自动判，其余转复核")。
4. **回归门 + 替换那 65 张**：以后每版新模型都必须在这批锚点上不掉点，才能上线。

(拿到数据我写个评估脚本：输入图 + 这份 CSV，输出各层准确率/置信区间/混淆矩阵/可靠性曲线/弱标签噪声率。)

## 五、边界(必须说清，别夸大)

这批只能测**良性、分布内**的精度。它是同事手上机器的凸包，**不是生产分布抽样、也没有对抗尾部**：

- 缺老旧机型、地区定制 ROM → 那部分测不到；
- **季度会老化**：新 iPhone / 新系统 / 新支付宝版本一出就要补采；
- **对"对抗尾部(好假图)多准"几乎什么都不说**——那要靠**红队集**(自造假图，给下界，已在做 `redteam_resolution_attack`)和**下游确认欺诈结果**(唯一真的对抗真值，延迟但真)。

所以"锚点集测良性精度 + 红队集测对抗下界 + 下游欺诈闭环"三条一起，才是完整的"知道自己多准"。

## 六、给经理的一句话

> 这是在**不做人工复核**的前提下，唯一能真正知道识别率的办法：标签来自"这台机器是什么系统"(采集，不是判图)，成本是几个人几天点机、不是标 12 万。有了它，识别率才是个能拍着胸脯说的数，而不是"大概 97% 吧"。

## 七、落地清单

1. 凑机器：自己的 + 同事的，尽量覆盖 iOS 各机型 + 安卓各厂商。
2. 每台按第二节的层截图，边截边在 CSV 里记一行。
3. 部分再拍屏一遍(翻拍层)。
4. 图 + CSV 打包给我 → 我跑评估，出各层准确率 + 置信区间 + 阈值标定。

## 八、今天就能起步(先跑通，再铺量)

不用等凑齐 1600。**今天先来一小批，把流程跑通**：

- 自己手机 +（能拉到的话）1–2 个同事，各截 5–10 张"转账成功"页，iOS / 安卓都要，边截边记进第三节那份 CSV。
- **30–50 张就够**我先把评估脚本写出来、验证 CSV 格式对得上、跑出第一版"预测系统 vs 真实系统"的准确率与混淆。
- 流程通了、格式确认了，再按第二节铺到 ~1600。

先跑通比先铺量重要：早发现"CSV 少列了机型""翻拍图忘标 photo"这类问题，比攒了 1600 张才返工强得多。这一小批本身也已经能给出设备识别的**第一个非循环准确率数字**。
